# 考试功能端到端测试与成绩发布修复报告（2026-03-01）

## 一、测试目标
- 验证学生考试作答、交卷、教师审核、成绩发布全链路可用。
- 验证“交卷超时/失败”与“成绩无法发布”问题是否已修复。

## 二、关键修复点

### 1）交卷接口稳定性优化
- 文件：`source/server/app/views/exam_views.py`
- 修复内容：
  - 交卷改为覆盖式提交（先清理旧答案，再写入新答案），避免重复提交造成答案堆积与性能下降。
  - 客观题即时评分；主观题默认待审核（避免同步 AI 评分阻塞请求）。

### 2）交卷后状态修复（影响成绩发布）
- 文件：`source/server/app/views/exam_views.py`
- 修复内容：
  - 学生交卷后，`ExamLogs.status` 从错误的 `2` 修正为 `1`（待发布）。
  - 教师发布成绩后再进入 `2`（已发布）。

### 3）教师成绩页 token 容错增强
- 文件：`source/client/src/views/pages/teacherExamLogs.vue`
- 修复内容：
  - `qryForm.token` 增加 `sessionStorage` 兜底。
  - `mounted` 时统一回填 token，避免刷新后列表为空、无法发布。

## 三、端到端实测记录（容器环境）

### 测试环境
- 后端容器：`fyp_backend`
- 数据库容器：`fyp_mysql`
- 前端容器：`fyp_frontend`

### 场景 A（含主观题）
1. 学生交卷（含填空+判断）。
2. 教师检查未审核状态（应提示存在待审核主观题）。
3. 教师审核填空题。
4. 再次检查（应提示审核完成）。
5. 教师发布成绩。

结果：
- 交卷返回 `code=0`。
- 审核前 `check` 返回 `flag=true`。
- 审核后 `check` 返回 `flag=false`。
- 发布成功后 `ExamLogs.status: 1 -> 2`，分数正确更新。

### 场景 B（仅客观题）
1. 学生交卷（仅判断题）。
2. 教师直接发布成绩。

结果：
- 交卷返回 `code=0`。
- `check` 返回可发布。
- 发布成功后 `ExamLogs.status: 1 -> 2`。

### 覆盖式提交验证
- 同一考试重复提交两次后：答案条数保持恒定，不再累积增长。
- 结论：覆盖式提交生效，避免了因历史堆积导致的慢请求与超时。

## 四、验证结论
- 学生端：切题保存、交卷、退出交卷流程可用。
- 教师端：审核、发布成绩流程可用。
- 状态流转：`考试中(0)` → `待发布(1)` → `已发布(2)` 正常。
- 性能：`/api/answerlogs/add/` 在测试中毫秒级到亚秒级，未再出现之前的超时失败链路。

## 五、建议的验收步骤（前端人工）
1. 学生账号进入考试，作答后交卷。
2. 教师账号进入“教师考试记录”，确认该学生记录状态为“待发布”。
3. 对主观题执行审核后，点击“公布成绩”。
4. 学生在“我的记录”查看分数与状态。
