# 答辩现场应急预案卡（30秒排错版）

## 使用原则
- 先稳住：先说“我先做快速健康检查”。
- 先保核心：优先保证登录、考试列表、练习列表、AI评分接口四链路。
- 先可解释：不能当场修完，也要说明原因与替代演示路径。

---

## 场景1：前端页面打不开

### 快速动作（30秒）
1. 确认后端是否正常：

```bash
cd source/server
python manage.py check
```

2. 执行一键校验脚本：

```bat
tools\run_defense_demo_check.bat
```

3. 若脚本通过，说明后端核心链路正常，切换“接口演示”模式。

### 现场话术
- “前端页面当前环境有显示问题，但后端核心链路已通过自动校验，我先演示系统可用性与回归结果。”

---

## 场景2：`/api/health/` 返回 500

### 可能原因
- 本机 MySQL 未启动
- AI API Key 未配置

### 快速动作
- 继续使用默认非严格模式（`/api/health/simple/` 必须通过）
- 若老师要求严格模式，说明依赖条件并现场补充

```bash
STRICT_HEALTH=1 python tools/run_defense_demo_check.py
```

### 现场话术
- “完整健康检查依赖数据库和外部 AI 服务。当前环境依赖未就绪，因此我采用非严格模式验证主链路稳定性。”

---

## 场景3：数据库连接失败

### 快速动作
1. 说明当前环境状态，切到回归基线（测试使用内存 SQLite）：

```bash
python tools/run_regression_baseline.py
```

2. 展示四链路通过结果，证明功能正确性与工程可验证性。

### 现场话术
- “数据库连接是环境问题，不是功能回归。回归基线在内存数据库下已完整通过，说明代码主链路正确。”

---

## 场景4：接口偶发超时/变慢

### 快速动作
1. 重试一次同接口。
2. 说明已做短缓存优化的接口：
   - `/api/exams/page/`
   - `/api/practicepapers/page/`

### 现场话术
- “我已对高频分页接口做短缓存，目标是提升演示场景稳定性。偶发抖动通常是现场环境资源波动。”

---

## 场景5：老师追问“为什么不做更重的安全/重构？”

### 回答模板
- “我采用分阶段策略：先做高收益低风险项，保证可交付与可验证，再逐步推进深度重构。”
- “当前版本重点是最小安全基线 + 自动回归基线，确保答辩和交付稳定。”

---

## 最终兜底演示路径（必过）

1. 运行：

```bat
tools\run_defense_demo_check.bat
```

2. 展示：
- Django check 通过
- 四链路回归 4/4 通过
- health/simple 通过

3. 口头总结：
- “系统具备可运行、可回归、可交付能力，符合 FYP 阶段目标。”
